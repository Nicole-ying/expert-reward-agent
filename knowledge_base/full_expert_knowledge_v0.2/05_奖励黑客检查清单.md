# 05_奖励黑客检查清单

## 1. 使用目的

奖励黑客是指智能体利用奖励函数漏洞获得高奖励，但没有完成真实目标。

每次 reward 生成后，必须进行奖励黑客检查。Agent 应扮演两个角色：

```text
Reward Designer：提出奖励函数
Reward Critic：质疑奖励函数
```

---

## Check 1：不完成任务也拿高分

```yaml
check_id: high_reward_without_success
问题: 智能体是否可以不触发 success，却通过辅助奖励拿到高 return？
需要日志:
  - success_rate
  - reward_terms.success
  - reward_terms.progress
  - reward_terms.alive
  - reward_terms.intrinsic
判断:
  - 如果 success_reward 很低但总 reward 很高，则风险高
修正:
  - 提高 terminal_success_reward
  - 降低辅助奖励
  - 加 success_rate 作为模型选择指标
```

---

## Check 2：原地不动刷分

```yaml
check_id: stand_still_exploit
问题: 智能体是否能原地不动获得较高 reward？
需要测试:
  - zero_action_policy
需要日志:
  - total_reward
  - reward_terms.alive
  - reward_terms.energy
  - action_magnitude
判断:
  - 如果 zero_action_policy return 接近训练策略 return，则风险高
修正:
  - 降低 alive_bonus
  - 降低过强 energy_penalty
  - 加 progress_delta_reward
  - 加 time_penalty
```

---

## Check 3：目标附近震荡刷分

```yaml
check_id: goal_oscillation_exploit
问题: 智能体是否能在目标附近来回移动刷 progress？
需要日志:
  - distance_to_goal
  - reward_terms.progress
  - success_flag
  - episode_length
判断:
  - 如果距离接近目标但 success_rate 低，且 progress 波动频繁，则风险高
修正:
  - 加 terminal_success_reward
  - 加 time_penalty
  - progress clip
  - 成功后终止 episode
```

---

## Check 4：猛冲后失败仍高分

```yaml
check_id: fast_then_fail_exploit
问题: 智能体是否能先拿大量前进奖励，然后摔倒/碰撞？
需要日志:
  - reward_terms.forward
  - failure_count
  - episode_length
  - action_magnitude
判断:
  - 如果 episode 短、失败多、forward reward 高，则风险高
修正:
  - 降低 forward 权重
  - 增加 fall/collision penalty
  - 加 stability_penalty
  - 加 action_smoothness_penalty
```

---

## Check 5：动作惩罚导致不动

```yaml
check_id: action_penalty_freeze
问题: 能耗或平滑惩罚是否导致智能体不敢动？
需要日志:
  - action_magnitude
  - reward_terms.energy
  - reward_terms.smoothness
  - progress
判断:
  - 如果 action 很小且 energy/smoothness 惩罚占主导，则风险高
修正:
  - 降低 action penalty
  - 增强 progress reward
  - 延迟引入动作惩罚
```

---

## Check 6：安全被效率奖励抵消

```yaml
check_id: safety_tradeoff_exploit
问题: 智能体是否为了速度/进度牺牲安全？
需要日志:
  - reward_terms.progress
  - reward_terms.safety
  - collision_count
  - failure_rate
判断:
  - 如果 return 不低但安全事件频繁，则风险高
修正:
  - 提高 safety penalty
  - 使用 gated_reward
  - 使用硬约束或终止
```

---

## Check 7：事件奖励重复刷分

```yaml
check_id: event_reward_farming
问题: 智能体是否能重复触发同一事件奖励？
需要日志:
  - event_trigger_count
  - reward_terms.event
  - stage_id
判断:
  - 如果同一事件触发过多但任务未完成，则风险高
修正:
  - 事件奖励只触发一次
  - 使用阶段状态机
  - 增加最终成功奖励
```

## 最低成本测试

每个新 reward 至少执行：

```yaml
mandatory_tests:
  - random_policy_reward_test
  - zero_action_policy_reward_test
  - failure_trajectory_reward_test
  - reward_terms_scale_test
```

| 测试 | 目的 |
|---|---|
| random_policy_reward_test | 检查随机策略是否异常高分 |
| zero_action_policy_reward_test | 检查原地不动是否刷分 |
| failure_trajectory_reward_test | 检查失败轨迹是否高分 |
| reward_terms_scale_test | 检查某个奖励项是否量级过大 |
