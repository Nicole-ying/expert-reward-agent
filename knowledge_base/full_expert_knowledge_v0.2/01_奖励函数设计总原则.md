# 01_奖励函数设计总原则

## 1. 奖励函数的本质

奖励函数不是任务本身，而是任务真实目标的代理表达。

真实目标：

```math
G_{\text{true}}
```

代理奖励：

```math
R_{\text{proxy}}
```

强化学习优化的是：

```math
\max_{\pi} \mathbb{E}_{\pi}\left[\sum_{t=0}^{T} \gamma^t R_{\text{proxy}}(s_t,a_t,s_{t+1})\right]
```

但研究者真正关心的是：

```math
G_{\text{true}} \text{ 是否被满足}
```

因此，奖励设计的核心风险是：

```text
R_proxy 被最大化，但 G_true 没有被满足。
```

这类问题可以表现为 reward hacking、specification gaming、proxy objective misalignment。

---

## 2. 专家设计奖励函数的根本问题

专家在设计奖励函数时，本质上是在回答五个问题：

```text
1. 智能体最终应该完成什么？
2. 智能体中途怎样才算朝目标变好？
3. 哪些行为绝对不能发生？
4. 哪些行为虽然能完成任务但不理想？
5. 奖励函数是否可能被智能体钻空子？
```

对应到奖励结构：

```math
R_t =
R_{\text{task}}
+
R_{\text{guide}}
-
C_{\text{safety}}
-
C_{\text{behavior}}
+
R_{\text{explore}}
```

其中：

| 符号 | 含义 |
|---|---|
| \(R_{\text{task}}\) | 任务目标奖励 |
| \(R_{\text{guide}}\) | 过程引导奖励 |
| \(C_{\text{safety}}\) | 安全约束惩罚 |
| \(C_{\text{behavior}}\) | 动作代价、平滑性、能耗等行为正则 |
| \(R_{\text{explore}}\) | 探索奖励 |

---

## 3. 先定义真实目标，再设计奖励函数

错误方式：

```text
先看环境变量，再拼一个 reward。
```

专家方式：

```text
先写真实目标 → 拆成目标/约束/偏好 → 再选奖励骨架。
```

示例：

```text
真实目标：
机器人应该稳定、快速、低能耗地向前走，并且不能摔倒。

拆解：
主目标：向前走
约束：不能摔倒
偏好：动作稳定、能耗低

对应奖励：
前进奖励 + 存活奖励 - 摔倒惩罚 - 能耗惩罚 - 动作平滑惩罚
```

---

## 4. 奖励项必须有明确角色

每个奖励项必须属于下列角色之一。

| 角色 | 作用 | 示例 |
|---|---|---|
| 任务目标奖励 | 表达最终目标 | 成功奖励、到达终点 |
| 过程引导奖励 | 缓解稀疏奖励，引导学习 | 距离奖励、增量奖励 |
| 安全约束惩罚 | 避免危险或非法行为 | 碰撞、摔倒、越界 |
| 行为正则惩罚 | 控制动作质量 | 能耗、动作平滑、姿态稳定 |
| 探索奖励 | 鼓励发现新状态 | 计数奖励、好奇心奖励 |
| 结构性奖励 | 控制奖励组合逻辑 | 门控、层级、字典序 |

如果一个奖励项无法说明角色，则不应加入。

---

## 5. 简单任务用简单奖励，复杂任务才用组合奖励

### 5.1 简单任务

如果任务满足：

```text
目标单一
成功/失败清楚
探索不难
动作空间简单
安全约束少
```

则可以从单一奖励开始：

```math
r_t = r_{\text{success}} - r_{\text{failure}}
```

或：

```math
r_t = r_{\text{alive}}
```

### 5.2 复杂任务

如果任务满足：

```text
多目标
连续控制
稀疏奖励
安全约束
动作质量要求
```

则使用组合奖励：

```math
r_t =
\sum_i w_i r_i
```

但组合奖励必须记录每个 reward term，否则无法诊断。

---

## 6. 从简单到复杂的奖励复杂度阶梯

```text
Level 1：单一奖励
Level 2：终点奖励 + 时间/存活项
Level 3：终点奖励 + 过程引导
Level 4：目标 + 过程 + 安全 + 动作正则
Level 5：复杂复合奖励，包括势能塑形、探索奖励、动态权重、多目标结构
```

专家一般不会一上来写 Level 5，而是从能工作的简单奖励开始。

---

## 7. 主任务奖励不能被辅助奖励淹没

如果：

```math
\sum_t R_{\text{alive}} \gg R_{\text{success}}
```

智能体可能只想活着，不愿完成任务。

如果：

```math
\sum_t R_{\text{progress}} \gg R_{\text{success}}
```

智能体可能只刷进度，不触发成功。

因此必须分析 reward_terms 的累计贡献。

---

## 8. 安全约束不能随意被抵消

如果安全是硬约束，不应该只用简单小惩罚：

```math
r = r_{\text{speed}} - 0.1 c_{\text{collision}}
```

因为高速奖励可能抵消碰撞惩罚。

更合理方式：

```math
r_t =
\begin{cases}
-R_{\text{large}}, & \text{unsafe}\\
r_{\text{task}}, & \text{safe}
\end{cases}
```

或者使用约束优化思想：

```math
\max_{\pi} J_{\text{task}}(\pi)
\quad
\text{s.t.}
\quad
J_{\text{safety}}(\pi) \leq \epsilon
```

---

## 9. 奖励塑形要区分“引导学习”和“改变目标”

普通 dense reward 可能改变原任务目标。

势能塑形的经典形式：

```math
F(s,a,s') = \gamma \Phi(s') - \Phi(s)
```

总奖励：

```math
R'(s,a,s') = R(s,a,s') + F(s,a,s')
```

该思想来自 potential-based reward shaping，核心价值是：在特定条件下加入额外引导而保持最优策略不变。

---

## 10. 奖励提高不等于任务成功

训练过程中必须同时观察：

```text
mean_return
success_rate
failure_rate
episode_length
真实任务指标
reward_terms
动作统计
轨迹行为
```

如果：

```text
mean_return 上升，但 success_rate 不上升
```

则可能存在奖励错设。

---

## 11. 每个 reward 都要经过奖励黑客检查

每次 reward 生成后必须问：

```text
1. 智能体能否不完成任务也拿高分？
2. 智能体能否原地不动刷分？
3. 智能体能否目标附近震荡刷分？
4. 智能体能否猛冲拿短期奖励后失败？
5. 辅助奖励是否压过主任务？
6. 动作惩罚是否导致不动？
```

奖励函数不是写完就相信，而是写完就怀疑。

---

## 12. 专家闭环

```text
设计 reward
→ 训练
→ 记录 reward_terms
→ 看行为
→ 诊断失败模式
→ 修改 reward
→ 检查 reward hacking
→ 做消融
→ 写入经验记忆
```

该闭环是 Reward Design Agent 的核心。
